# UX 라이팅

## 기본 톤
- 사용자에게 말하듯 쓴다. 시스템 내부 용어(예: "422 에러", "null 값")를 그대로 노출하지 않는다.
- 짧고 능동적인 문장을 쓴다. "저장이 완료되지 않았습니다" 보다 "저장하지 못했어요" 처럼 사람이 말하듯 쓴다.
- 같은 개념에는 화면 전체에서 같은 용어를 쓴다 (예: "삭제"와 "제거"를 섞어 쓰지 않는다).

## 버튼 & 액션 라벨
- 버튼 텍스트는 클릭했을 때 무슨 일이 일어나는지 구체적으로 담는다. "확인" 보다는 "삭제하기", "저장하기"처럼
  동사+대상을 명시한다.
- 파괴적 액션 버튼에는 되돌릴 수 없다는 것을 암시하는 문구를 함께 쓴다 (예: "영구 삭제").
- 로딩 중에는 버튼 텍스트를 상태에 맞게 바꾼다 ("저장하기" → "저장 중…").

## 에러 메시지
- 무엇이 잘못됐는지 + 어떻게 고칠 수 있는지를 함께 알려준다. "오류가 발생했습니다"만으로 끝내지 않는다.
  예: "이메일 형식이 올바르지 않아요. example@domain.com 형태로 입력해 주세요."
- 사용자 잘못이 아닌 시스템 오류(서버 에러 등)는 사용자를 탓하는 어투를 쓰지 않는다. "다시 시도해 주세요"처럼
  다음 행동을 제시한다.
- 에러 메시지에 스택 트레이스, 내부 코드, 디버그 정보를 그대로 노출하지 않는다 (개발 모드 제외).

## 빈 상태 & 온보딩
- "데이터 없음" 대신 왜 비어있는지와 무엇을 하면 채워지는지 안내한다.
  예: "아직 프로젝트가 없어요. 첫 프로젝트를 만들어보세요." + CTA 버튼.

## 확인/경고 다이얼로그
- 되돌릴 수 없는 액션(삭제, 초기화)에는 무엇이 삭제되는지 구체적으로 명시한다
  ("정말 삭제하시겠어요?" 보다 "'프로젝트 A'를 삭제하면 되돌릴 수 없어요.").
- 확인 버튼 라벨은 "확인" 대신 실제 액션명을 쓴다 ("삭제", "초기화").

## 흔한 실수
- 개발자에게는 익숙하지만 사용자는 모르는 용어를 그대로 노출 (예: "payload", "엔드포인트").
- 모든 알림을 "성공했습니다" / "실패했습니다"로만 통일해서 무엇에 대한 결과인지 불명확함.
- 같은 의미의 버튼이 화면마다 다른 이름으로 표시됨 (예: 어떤 화면은 "취소", 다른 화면은 "닫기").

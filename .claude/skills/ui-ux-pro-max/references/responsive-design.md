# 반응형 / 적응형 디자인

## 기본 전략
- 모바일 우선(mobile-first)으로 작성하고, 브레이크포인트마다 필요한 만큼만 레이아웃을 확장한다.
  데스크톱을 먼저 만들고 모바일에서 억지로 욱여넣지 않는다.
- 대표 브레이크포인트 예시: ~375px(소형 모바일), ~768px(태블릿), ~1024px(소형 데스크톱), ~1440px(대형 데스크톱).
  프로젝트에 이미 정의된 브레이크포인트가 있다면 그것을 따른다.
- 뷰포트 폭이 아니라 콘텐츠가 깨지는 지점을 기준으로 브레이크포인트를 잡는 것이 더 견고하다.

## 레이아웃 전환
- 좁은 화면에서는 다단(multi-column) 레이아웃을 단일 컬럼으로 쌓는다. 그리드/플렉스의 `wrap`을 활용해서
  미디어 쿼리 없이도 자연스럽게 흐르게 만들 수 있는지 먼저 검토한다.
- 데스크톱의 사이드바/테이블은 모바일에서 하단 탭바, 카드 리스트, 아코디언 등 다른 패턴으로 재구성하는 것을
  고려한다. 단순 축소만으로는 터치 사용성이 나빠진다.
- 터치 타겟(버튼, 링크, 체크박스)은 최소 44x44px(iOS 기준) / 48x48px(Android 기준) 크기를 확보한다.

## 이미지 & 미디어
- 이미지는 `max-width: 100%`로 컨테이너를 넘치지 않게 하고, 가능하면 `srcset`/`sizes`로 뷰포트에 맞는
  해상도를 제공한다.
- 고정 픽셀 크기의 아이콘/이미지가 고밀도(retina) 화면에서 흐려 보이지 않는지 확인한다.

## 컨텐츠 우선순위
- 좁은 화면에서는 모든 요소를 다 보여줄 공간이 없다. 핵심 액션/정보를 우선 노출하고, 부차적인 것은
  접거나(아코디언, "더보기") 다음 화면으로 넘긴다.
- 데스크톱에서만 보이는 hover 툴팁에 의존하는 정보는 모바일에서 접근할 다른 방법(탭하면 표시 등)을 마련한다.

## 흔한 실수
- 표(table)를 모바일에서 가로 스크롤 없이 그대로 욱여넣어 글자가 깨짐.
- 데스크톱 hover 메뉴가 모바일에서는 아예 열리지 않음.
- 가로/세로 방향 전환(orientation change) 시 레이아웃이 깨지는 것을 테스트하지 않음.

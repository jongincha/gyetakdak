# 인터랙션 & 모션

## 상태 피드백
- 클릭/탭 가능한 요소는 hover, active, focus, disabled 상태를 모두 정의한다. hover만 있고 나머지가
  기본값 그대로인 경우가 흔하다.
- 비동기 액션(저장, 제출, 삭제)은 반드시 로딩 상태를 보여준다. 버튼을 눌렀는데 아무 반응이 없으면
  사용자는 다시 누르고, 중복 요청이 발생할 수 있다.
- 성공/실패 결과는 명시적으로 알려준다 (토스트, 인라인 메시지 등). 조용히 실패하는 UI가 가장 나쁘다.

## 트랜지션
- 트랜지션 지속시간은 100~300ms를 기본으로 한다. 마이크로 인터랙션(hover, 토글)은 짧게(100~150ms),
  화면 전환/모달 등장은 조금 더 길게(200~300ms).
- `ease-in-out` 또는 `cubic-bezier` 기반의 자연스러운 이징을 사용한다. `linear`는 기계적으로 느껴져서
  UI 모션에는 잘 쓰지 않는다.
- 애니메이션은 의미가 있어야 한다: 요소가 어디서 왔다가 어디로 가는지, 무엇이 무엇으로 바뀌는지 사용자가
  따라갈 수 있게 돕는 용도로 쓴다. 장식만을 위한 과도한 모션은 지양한다.

## 접근성과 모션
- `prefers-reduced-motion: reduce` 미디어 쿼리를 존중해서, 모션에 민감한 사용자에게는 트랜지션을
  최소화하거나 즉시 전환되게 한다.
- 자동 재생되는 캐러셀/슬라이드는 일시정지 컨트롤을 제공한다.

## 로딩/빈 상태/에러 상태
- 데이터가 없을 때(빈 상태), 로딩 중일 때, 에러가 났을 때 각각 별도의 UI를 설계한다. "데이터가 있을 때"만
  디자인하고 나머지는 즉흥적으로 처리하지 않는다.
- 스켈레톤 로딩은 실제 콘텐츠의 레이아웃과 비슷한 형태로 만들어서 로딩 후 레이아웃이 크게 점프하지 않게 한다.
- 빈 상태에는 "왜 비어있는지"와 "다음에 뭘 할 수 있는지"를 함께 안내한다 (단순히 "데이터 없음"만 두지 않는다).

## 흔한 실수
- 더블클릭/연타로 동일 요청이 중복 전송됨 (제출 중 버튼 비활성화 누락).
- 모든 트랜지션에 동일한 500ms+ duration을 적용해서 UI가 굼떠 보임.
- 에러 발생 시 콘솔에만 로그를 남기고 사용자에게는 아무 표시도 하지 않음.

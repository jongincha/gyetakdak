# 접근성 (Accessibility)

## 시맨틱 마크업
- `div`/`span`으로 모든 걸 만들지 않는다. 버튼은 `<button>`, 링크는 `<a href>`, 제목은 `<h1>~<h6>`을
  실제 문서 구조에 맞게 사용한다.
- 제목 레벨을 시각적 크기 때문에 건너뛰지 않는다 (h1 다음 바로 h4로 가지 않기). 시각 스타일은 CSS로 조정한다.
- 폼 입력에는 반드시 연결된 `<label>`이 있어야 한다 (placeholder는 label 대체품이 아니다).

## 키보드 접근성
- 모든 인터랙티브 요소(버튼, 링크, 입력, 커스텀 드롭다운/모달)는 Tab으로 도달 가능하고, Enter/Space로
  실행 가능해야 한다.
- 포커스 순서(tab order)는 시각적 순서와 일치해야 한다. `tabindex`에 양수를 넣어 순서를 억지로 바꾸지 않는다.
- 모달/드롭다운을 열면 포커스를 그 안으로 이동시키고, Esc로 닫히며, 닫힌 뒤 포커스는 트리거 요소로 복귀한다.
- 포커스 표시(`:focus-visible`)를 임의로 제거하지 않는다. 디자인상 기본 아웃라인이 마음에 안 들면 반드시
  대체 스타일(윤곽선, 배경색 변화 등)을 제공한다.

## 스크린리더
- 아이콘만 있는 버튼에는 `aria-label`로 목적을 설명한다 (예: "닫기", "메뉴 열기").
- 장식용 이미지/아이콘은 `aria-hidden="true"` 또는 빈 `alt=""`로 스크린리더가 건너뛰게 한다.
- 상태 변화(에러 메시지 등장, 로딩 완료, 저장됨 알림)는 `aria-live` 영역으로 알려서 화면을 보지 않아도
  인지할 수 있게 한다.
- 커스텀 컴포넌트(탭, 아코디언, 콤보박스)는 가능하면 네이티브 HTML 요소나 검증된 헤드리스 라이브러리
  (Radix, React Aria 등)를 사용한다. 처음부터 ARIA를 직접 구현하는 것은 실수하기 매우 쉽다.

## 폼 접근성
- 에러 메시지는 해당 입력 필드 바로 옆에, 텍스트로 명확히 표시한다 (테두리 색만 빨갛게 바꾸는 것으로 끝내지 않는다).
- 필수 입력 여부는 `required`/`aria-required`와 함께 시각적으로도(별표 등) 표시한다.
- 유효성 검사 에러는 제출 시점뿐 아니라, 가능하면 필드를 벗어날 때(blur) 알려준다.

## 흔한 실수
- `onClick`만 있고 키보드 핸들러가 없는 `<div>` 버튼.
- 대비를 위해서만 존재하는 장식 아이콘에 불필요한 `aria-label`을 넣어 스크린리더가 중복으로 읽음.
- 모달을 열었는데 배경 콘텐츠가 여전히 Tab으로 접근 가능함 (포커스 트랩 누락).

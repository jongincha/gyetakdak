# UI-UX Pro Max

화면 하나를 찍어내듯 만드는 게 아니라, 하나의 디자인 시스템처럼 일관되게 보이도록 만들기 위한 규칙 모음입니다.
새 UI를 만들거나 기존 UI를 고칠 때, 아래 순서로 작업하세요.

## 작업 순서

1. **요구사항 파악** — 어떤 화면/컴포넌트인지, 주 사용자와 핵심 태스크가 무엇인지 한 문장으로 정리한다.
2. **참고 문서 로드** — 아래 "규칙 카테고리"에서 관련된 `references/*.md`를 열어 세부 체크리스트를 확인한다.
   전체를 다 읽을 필요는 없고, 지금 작업과 관련된 파일만 골라 읽는다.
3. **설계/구현** — 체크리스트를 지키며 마크업·스타일·인터랙션을 작성한다.
4. **자체 검수** — 맨 아래 "완료 기준(Definition of Done)"으로 스스로 점검한다.
5. **실제로 확인** — 가능하면 브라우저(또는 시뮬레이터)에서 실제로 렌더링해보고, 라이트/다크 모드와 최소 2개 뷰포트
   (모바일/데스크톱)에서 확인한다. 코드가 "그럴듯해 보인다"는 것과 "실제로 그렇게 보인다"는 것은 다르다.

## 핵심 원칙

- **일관성 우선**: 새로운 색상·간격·폰트 크기를 즉흥적으로 만들지 말고, 기존 디자인 토큰/스케일을 재사용한다.
  토큰이 없다면 이번 작업에서 만드는 값이 이후 기준이 된다는 것을 의식하고 스케일에 맞게 정의한다.
- **콘텐츠가 먼저, 장식은 그다음**: 레이아웃과 컴포넌트는 실제 콘텐츠(긴 텍스트, 빈 상태, 에러 상태, 다국어)를
  기준으로 설계한다. 로렘입숨이나 이상적인 데이터로만 검증하지 않는다.
- **접근성은 선택이 아니다**: 색만으로 정보를 구분하지 않고, 키보드만으로 전체 플로우를 완료할 수 있어야 하며,
  스크린리더 사용자도 같은 정보를 얻을 수 있어야 한다.
- **과설계 금지**: 요청받지 않은 애니메이션, 테마 전환, 커스텀 컴포넌트 추상화를 미리 만들지 않는다.
  지금 화면에 필요한 만큼만 만든다.
- **되돌릴 수 있는 선택**: 디자인 결정(색상 팔레트, 컴포넌트 구조)이 다른 화면에도 영향을 준다면, 왜 그렇게
  했는지 한 줄로 남기되, 장황한 설계 문서는 만들지 않는다.

## 완료 기준 (Definition of Done)

작업을 끝내기 전에 아래를 스스로 확인합니다. 하나라도 "아니오"면 작업이 끝난 게 아닙니다.

- [ ] 간격·폰트 크기·색상이 기존 스케일/토큰에서 나온 값인가? (임의의 매직 넘버가 없는가)
- [ ] 텍스트와 배경의 명도 대비가 WCAG AA(본문 4.5:1, 큰 텍스트 3:1) 이상인가?
- [ ] 모든 인터랙티브 요소가 키보드(Tab/Enter/Space/Esc)만으로 조작 가능한가?
- [ ] 포커스 상태가 시각적으로 명확히 보이는가? (`outline: none`만 지우고 대체하지 않은 경우 없는가)
- [ ] 로딩/빈 상태/에러 상태/긴 텍스트 등 엣지 케이스를 실제로 렌더링해서 확인했는가?
- [ ] 라이트/다크 테마 모두에서 깨지지 않는가? (하드코딩된 흰색/검정 배경 없는가)
- [ ] 최소 2개 뷰포트(예: 375px 모바일, 1280px 데스크톱)에서 레이아웃이 무너지지 않는가?
- [ ] 이미지/아이콘 전용 버튼에 접근 가능한 이름(`aria-label` 등)이 있는가?
- [ ] 애니메이션이 `prefers-reduced-motion`을 존중하는가? (의미 전달용이 아닌 순수 장식 모션이 있다면)
